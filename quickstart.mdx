---
title: 'Quickstart'
description: 'Get the Literacy Platform running locally in a few minutes.'
---

<Steps>
  <Step title="Check prerequisites">
    You will need the following before starting:

    - **Node.js 20+** — the project uses Next.js 16 and requires Node 20 or later.
    - **PostgreSQL** — a running instance (local or remote). The Prisma schema targets PostgreSQL.
    - **Cloudflare R2** — a bucket and API credentials for file storage. You will need an R2 endpoint URL, access key ID, secret access key, bucket name, and public URL.
    - A terminal with `npm` and `npx` available.
  </Step>

  <Step title="Clone and install dependencies">
    Clone the repository and install Node dependencies.

    <CodeGroup>

    ```bash npm
    git clone <repository-url> literacy-platform
    cd literacy-platform
    npm install
    ```

    ```bash yarn
    git clone <repository-url> literacy-platform
    cd literacy-platform
    yarn install
    ```

    ```bash pnpm
    git clone <repository-url> literacy-platform
    cd literacy-platform
    pnpm install
    ```

    </CodeGroup>
  </Step>

  <Step title="Configure environment variables">
    Create a `.env` file in the project root. All variables below are required at runtime.

    ```bash .env
    # PostgreSQL connection string
    DATABASE_URL="postgresql://user:password@localhost:5432/literacy_platform"

    # Shared secret used to sign both admin and student JWTs (HS256)
    JWT_SECRET="replace-with-a-long-random-secret"

    # Cloudflare R2 — S3-compatible endpoint for your account
    R2_ENDPOINT="https://<account-id>.r2.cloudflarestorage.com"
    R2_ACCESS_KEY_ID="<your-r2-access-key-id>"
    R2_SECRET_ACCESS_KEY="<your-r2-secret-access-key>"
    R2_BUCKET_NAME="literacy-platform-content"

    # Public base URL for R2 objects served to students (admin content)
    R2_PUBLIC_URL="https://pub-<hash>.r2.dev"

    # Shared secret between the Cloudflare Worker and Next.js internal endpoints
    WORKER_SECRET="replace-with-a-long-random-secret"
    ```

    <Note>
      `JWT_SECRET` and `WORKER_SECRET` should be long, randomly generated strings. You can generate one with `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`.
    </Note>

    <Info>
      `R2_PUBLIC_URL` is only used for admin-uploaded content (`admin_content` upload context). Receipts and student audio are served via short-lived presigned GET URLs and are never exposed through the public URL.
    </Info>
  </Step>

  <Step title="Run database migrations and seed">
    Apply the Prisma schema to your PostgreSQL database and seed initial data.

    ```bash
    npx prisma migrate dev
    npx prisma db seed
    ```

    `prisma db seed` runs `prisma/seed.ts` via `ts-node` and creates the initial admin account and any default content configured in the seed file.

    <Note>
      If you are connecting to a remote or shared database, use `npx prisma migrate deploy` instead of `migrate dev` to apply migrations without generating new migration files.
    </Note>
  </Step>

  <Step title="Start the development server">
    ```bash
    npm run dev
    ```

    The Next.js app will start on [http://localhost:3000](http://localhost:3000) by default.

    <Info>
      Rate limiting is disabled automatically in development (`NODE_ENV=development`). All request limits (login, registration, presign, upload) are bypassed locally so they do not interfere with testing.
    </Info>
  </Step>

  <Step title="Access the admin and student portals">
    The platform exposes two protected portals, each guarded by edge middleware:

    | Portal | URL | Cookie |
    |--------|-----|--------|
    | Admin | [/admin/login](http://localhost:3000/admin/login) | `admin_token` |
    | Student | [/student/login](http://localhost:3000/student/login) | `student_token` |

    Log in with the admin credentials created during the seed step. Accessing any `/admin/*` or `/student/*` route without a valid session cookie will redirect to the respective login page.

    If a session cookie is present but the JWT has expired, the middleware redirects to the login page with `?expired=1` appended so the UI can display a "session expired" message.
  </Step>

  <Step title="Register a first student">
    Visit [/register](http://localhost:3000/register) to submit a student registration. The registration form collects parent details (first name, last name, email, phone) and child details (first name, last name, grade, date of birth).

    After submission, the student account enters the `pending_payment` state. An admin must then:

    1. Review the payment submission in the admin portal.
    2. Approve the payment to advance the student to `approved_pending_login`.
    3. Create login credentials for the child.
    4. Assign a literacy level after reviewing the initial assessment.

    See the [architecture overview](/architecture) for a full description of the account state machine.
  </Step>
</Steps>
